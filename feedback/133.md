If you focus purely on **performance**, the main bottleneck in your current solution is this step:

```elixir
Enum.uniq(nums)
|> Enum.sort(&>/2)
```

That combination forces:

* **O(n)** for `uniq` (with extra memory via a map internally)
* **O(n log n)** for sorting (dominant cost)
* extra intermediate list allocations

So overall: **O(n log n)** time + multiple full passes over the data.

## 🚀 Best performance improvement: single-pass scan (O(n), O(1) extra space)

Instead of building intermediate lists, you track:

* largest
* second largest (distinct)

### ✅ Optimal Elixir solution

```elixir
def find_second_largest(nil), do: nil
def find_second_largest([]), do: nil

def find_second_largest([first | rest]) do
  {max1, max2} =
    Enum.reduce(rest, {first, nil}, fn x, {max1, max2} ->
      cond do
        x > max1 ->
          {x, max1}

        x < max1 and (is_nil(max2) or x > max2) ->
          {max1, x}

        true ->
          {max1, max2}
      end
    end)

  max2
end
```

---

## 📊 Why this is faster

### Your version

* `uniq`: O(n)
* `sort`: O(n log n)
* multiple allocations (lists created twice)

### Optimized version

* single pass: O(n)
* constant memory: O(1)
* no sorting
* no intermediate lists

## 🧠 Subtle but important improvement detail

We explicitly enforce:

```elixir
x < max1
```

This ensures:

* duplicates of the max don’t incorrectly become second max
* no need for `Enum.uniq/1` at all

So we remove an entire traversal.

## 🧩 If you want a *slight upgrade* without full rewrite

You could also do a middle-ground optimization:

```elixir
nums
```

But this is actually worse in practice because:

* still O(n log n)
* still sorts everything
* just defers deduping

So it’s not a real win.
